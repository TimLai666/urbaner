# 輸出前自我檢查清單

完成報告草稿後，在交付給使用者前執行以下檢查。發現問題立即修正，不要交付有已知問題的報告。

---

## 第一關：事實查核

- [ ] 報告中每一個數字都有來源標注（融資金額、市場規模、退貨率、用戶數等）
- [ ] 沒有出現「大約」、「估計」、「可能」後面接未查證的數字
- [ ] 所有引言都有說話者身份 + 出處 + 日期
- [ ] 投資方清單已查證每個投資方是否真的投資了這家公司
- [ ] 沒有同名公司資料混入（確認所有資料都是針對正確的公司）
- [ ] 時間線的每個節點都有來源，且來源有衝突時已說明採信理由

---

## 第二關：分析鏈一致性

- [ ] 生態系地圖在 PESTEL 之前
- [ ] PESTEL 有優先分評分（重要性 × 影響程度 × 時效性）
- [ ] SWOT 的 O/T 對應 PESTEL 移交包，並標了 PESTEL 編號與優先分
- [ ] STP 的 S（區隔）只引用 PESTEL 優先分 ≥ 48 的因素，並明確說明低分因素不納入
- [ ] STP 的 T（目標）有三欄並列的篩選矩陣（PESTEL + 五力 + SWOT）
- [ ] STP 的 P（定位）只從主要策略推導，沒有同時引用多個未選中的策略
- [ ] BMC 的震央選擇有從 SWOT + 五力推導的表格依據，不是主觀選定
- [ ] Persona 的每個屬性都有來源依據
- [ ] 服務藍圖以 Persona 為主角，流程階段與 CJM 的旅程階段對應
- [ ] CJM 的情緒分數有真實用戶反饋支撐（不是假設）
- [ ] CJM 包含真實的摩擦點，不是品牌視角的理想化版本
- [ ] 背後啟發的每一點都有明確的分析來源（從哪個分析模型的哪個結論推導）

---

## 第三關：報告結構順序

正確順序（依序 ✓）：
- [ ] 案例名稱
- [ ] 公司背景
- [ ] 創辦人背景
- [ ] 推出這項服務的背景
- [ ] 推出時間（時間線）
- [ ] 推薦原因
- [ ] 應用情境
- [ ] 創新類型（六選一）
- [ ] 創新模式（複選）
- [ ] 產業生態系地圖
- [ ] PESTEL 分析
- [ ] 五力分析
- [ ] SWOT 分析
- [ ] SWOT 碰撞策略
- [ ] 主要策略選擇
- [ ] 鎖定客群（STP）
- [ ] 服務模式 / 商業模式圖（BMC）
- [ ] 顧客 Persona
- [ ] 服務藍圖
- [ ] 顧客旅程地圖（CJM）
- [ ] 創造效果 / 解決問題
- [ ] 相關展示（非必要）
- [ ] 目前成效 / 獲獎情況 / 外部討論關鍵
- [ ] 背後代表的啟發與意義
- [ ] **參考資料（最後，且是唯一的最後區段）**

---

## 第四關：參考資料

- [ ] 有獨立的 `## 參考資料` 區段
- [ ] 參考資料是報告的最後一個區段
- [ ] 參考資料之後沒有任何其他內容
- [ ] 所有來源都整合在參考資料區段，沒有只出現在 `> 來源：` 標注中而未進入參考資料清單的
- [ ] 沒有重複的編號
- [ ] 有按類型分類（媒體報導 / 投資方資料 / 創辦人一手來源 / 市場數據 / 用戶評論）

---

## 第五關：常見問題快速掃描

快速檢查以下在 Doji 案例中曾出現的具體錯誤：

- [ ] STP 定位沒有同時用多個策略（只用主要策略）
- [ ] Persona 不是憑空虛構的（每個屬性有來源）
- [ ] CJM 不是理想化版本（有真實批評和摩擦）
- [ ] 生態系地圖的競爭層有列出具名競爭者（不只有大廠）
- [ ] 商業模式的創造效果區段有三層（宣稱 / 實際 / 缺口），不只有正面效果
- [ ] 目前成效區段說明了哪些數字是無公開來源的
- [ ] 參考資料編號沒有重複（特別檢查是否有兩個「10」或「11」）

---

## 緊急修復指南

若發現以下問題，按照以下方式修復：

**問題：參考資料夾在其他區段中間**
修復：用 Python 腳本重新排列區段，不要手動剪貼（容易出錯）

**問題：某個區段的內容不見了**
修復：不要覆蓋整個文件，用 str_replace 在正確位置補回，或用 Python 腳本精確插入

**問題：分析鏈有環節缺失**
修復：在缺失的環節重新執行對應的分析，確認輸入輸出正確後寫入報告

**問題：Persona 屬性沒有來源**
修復：回到 `01-research-protocol.md` 的 Persona 查詢順序，重新查詢後補上來源或修改屬性
