# 分析鏈完整邏輯

## 總覽：分析鏈的流向

```
生態系地圖（了解所有利害關係人）
    ↓
PESTEL（外部環境掃描，各因素評分）
    ↓ [移交包：機會池＋威脅池，依優先分排序]
五力分析（競爭格局，切換成本，替代品）
    ↓
SWOT（S/W 來自公司自身，O/T 來自 PESTEL 高分因素）
    ↓ [碰撞 → SO/WO/ST/WT × 2 = 8個策略]
主要策略選擇（1個，有依據的理由）
    ↓
STP（S 用 PESTEL≥48 因素 + 五力；T 三軌篩選；P 只從主要策略推導）
    ↓
商業模式圖（震央從 SWOT+五力推導）
    ↓
顧客 Persona（從 STP 目標客群 + 有來源事實萃取）
    ↓
服務藍圖（以 Persona 為主角，六層結構）
    ↓
顧客旅程地圖（以 Persona 為主角，真實摩擦，情緒分數）
```

---

## 各分析層的輸入輸出規格

### 生態系地圖

**輸入**：公司名稱、業務描述
**核心問題**：這個服務的生態系裡，誰是供應方、誰是需求方、誰是競爭者、誰是監管者？
**輸出**：Mermaid 圖（分層結構）

**必要的層次（不能省略）：**
- 投資方（含輪次金額）
- 技術來源（創辦人前雇主、研究機構）
- 核心公司
- 上游合作方（品牌、內容供給方）
- 電商/導購目的地
- 通路（App Store、Web、社群）
- 直接競爭者（新創，含融資）
- 間接競爭者（大廠自建方案）
- 監管環境

**箭頭標準：**
- 實線 + 箭頭（→）：主動合作、資金流向、服務傳遞
- 虛線 + 箭頭（-.→）：間接影響（技術積累、過往合作）
- 無方向雙線（---）：競爭關係

---

### PESTEL 分析

**輸入**：公司背景、業務性質
**核心問題**：外部環境中，哪些宏觀因素在推動或阻礙這個業務？
**輸出**：評分表 + 各因素詳細說明 + SWOT 移交包

**評分系統：**
```
優先分 = 重要性(1-5) × 影響程度(1-5) × 時效性(1-3)
最高分：75（5×5×3）
有效因素門檻：≥12
進入 SWOT 的門檻：≥48（建議）
```

**六個面向的典型因素：**
- Political（政治）：監管政策、平台審查、進出口限制
- Economic（經濟）：市場規模、退貨成本、VC 投資趨勢
- Social（社會）：用戶行為轉變、社群文化、人口結構
- Technological（技術）：核心技術成熟度、大廠技術進場、開源替代
- Environmental（環境）：ESG、碳排、永續壓力
- Legal（法律）：AI 法規、隱私法、智財

**SWOT 移交包格式：**
```
機會池（O候選，依優先分排序）：
  [編號] 因素名稱｜優先分：X｜簡要說明
  ...

威脅池（T候選，依優先分排序）：
  [編號] 因素名稱｜優先分：X｜簡要說明
  ...

觀察項（未達門檻）：
  [編號] 因素｜升級條件：...
```

### 五力分析

**輸入**：PESTEL（競爭環境），生態系地圖（競爭者清單）
**核心問題**：這個產業的競爭結構是什麼？公司的議價空間在哪裡？
**輸出**：五力判斷（含等級）+ 競爭者清單

**五力評估維度：**

1. 現有競爭者的競爭強度
   - 直接競爭者（同類新創，有名稱有金額）
   - 間接競爭者（大廠自建方案）
   - 競爭強度等級：低/中低/中/中高/高

2. 新進者威脅
   - 技術門檻（是否可用開源替代）
   - 資本門檻
   - 現有玩家的護城河

3. 替代品威脅
   - 替代品的覆蓋範圍
   - 替代品無法做到的事（差異化空間）

4. 供應商議價能力
   - 核心技術依賴（是自研還是外採）
   - 內容/品牌合作方的可替代性

5. 買方議價能力
   - B2C 切換成本（低/中/高）
   - B2B 客戶的選擇多寡

**關鍵輸出**：買方切換成本的判斷（影響 BMC 震央選擇和 STP 定位）

---

### SWOT 分析

**輸入**：PESTEL 移交包，五力分析，公司調研
**核心問題**：公司的優劣勢是什麼？外部有什麼機會和威脅？
**輸出**：S1-S3、W1-W3、O1-O3、T1-T3

**配置規則：**
- S（優勢）：公司內部，有事實來源的競爭優勢
- W（劣勢）：公司內部，有事實來源的限制或不足
- O（機會）：來自 PESTEL 機會池，標 PESTEL 編號與優先分
- T（威脅）：來自 PESTEL 威脅池，標 PESTEL 編號與優先分

**策略碰撞格式：**
```
SO1 [S××O×]：名稱——說明
SO2 [S××O×]：名稱——說明
WO1 [W××O×]：名稱——說明
WO2 [W××O×]：名稱——說明
ST1 [S××T×]：名稱——說明（主要策略來自這裡）
ST2 [S××T×]：名稱——說明
WT1 [W××T×]：名稱——說明
WT2 [W×××T×]：名稱——說明
```

**主要策略選擇的依據標準：**
- 引用 PESTEL 最高優先分的威脅/機會
- 引用公司創辦人或媒體的真實話語作為佐證
- 解釋「為何這個策略比其他 7 個更強」

---

### STP 分析

**輸入**：PESTEL（只用優先分≥48）、五力、SWOT、主要策略
**核心問題**：應該服務誰？如何讓他們選擇我們而非競爭者？
**輸出**：S（區隔維度 + 四格矩陣）、T（三軌篩選矩陣）、P（定位句 + 競爭對照表）

**S（區隔）設計邏輯：**
- 每個區隔維度必須對應至少一個 PESTEL 因素（標優先分）
- 維度切割要能清楚分出不同的用戶群
- 二維交叉可產生 4 個區隔，三維交叉可產生 8 個

**T（目標）三軌篩選矩陣格式：**
| 區隔 | PESTEL（標因素編號與優先分） | 五力結構 | SWOT 能力 | 結論 |
|------|--------------------------|----------|-----------|------|
| AX | ... | ... | ... | ✅主要目標 |
| BX | ... | ... | ... | ⚠️次要 |
| AY | ... | ... | ... | ❌排除 |
| BY | ... | ... | ... | ❌排除 |

排除某個區隔時，**最強的排除依據**應該引用 PESTEL 優先分最高的威脅因素，不是主觀判斷。

**P（定位）推導邏輯：**
1. 確認主要策略是 [SxTy]
2. Sx = 我們的核心優勢（是什麼）
3. Ty = 我們面對的最大威脅（競爭者的強項是什麼）
4. 定位 = 在 Ty 的競爭場景中，Sx 讓我們站在哪個位置

**定位對照表**：只與主要策略的競爭對手比較（不列所有競爭者）

---

### 商業模式圖（BMC）

**輸入**：SWOT、五力、STP
**核心問題**：公司靠什麼創造、傳遞和捕獲價值？
**輸出**：震央選擇依據表 + 九格商業模式圖

**震央選擇邏輯：**

| 震央 | 選這個的條件 | 排除其他的條件 |
|------|------------|--------------|
| 資源導向 | S1/S2 是稀缺資源 + 主要策略從資源優勢出發 | 買方切換成本低（排除顧客導向）+ W3 商業化不明（排除財務導向）|
| 顧客導向 | 有清楚未滿足需求 + 切換成本高 + STP 以痛點出發 | — |
| 財務導向 | 收益模式清晰 + 成本結構是競爭優勢 | 商業化路徑未確立時排除 |
| 產品導向 | 產品/技術本身是唯一差異 + IP 保護強 | — |

**九格設計順序（資源導向為例）：**
關鍵資源★ → 價值主張★ → 目標客群 → 通路 → 顧客關係 → 關鍵活動 → 關鍵合作夥伴 → 收益流⏸ → 成本結構⏸

★ = 核心要素（深度設計）
⏸ = 暫緩要素（說明暫緩原因）

**必須包含的子模組：**
- 差異化策略地圖（假說 / 機制 / 防禦性 / 主要風險）
- 30 天驗證實驗（目標要素 / 方法 / 成功指標 / 決策規則）
- WT → SO 轉換路徑

### 顧客 Persona

**輸入**：STP 目標客群、有來源的 beta 用戶/實際用戶資料
**核心問題**：目標客群中的一個真實代表是誰？
**輸出**：Persona 屬性表格（每列有來源）+ 核心痛點 + 核心動力

**屬性來源查詢順序：**
1. 媒體報導中提到的具名用戶（最可信）
2. App Store / 評論平台的具體描述（有代表性的評論）
3. 創辦人對目標用戶的自述（二手，需標注）
4. 公司策展/合作內容暗示的客群（推論，需說明推論邏輯）

**不允許**：純粹虛構的 Persona 屬性（沒有任何來源依據）

---

### 服務藍圖

**輸入**：Persona、BMC 的服務傳遞分析
**核心問題**：服務在前台和後台是如何被傳遞的？哪裡有缺口？
**輸出**：六層 × N 個階段的表格 + 缺口診斷表

**六層結構（縱軸）：**
1. 實體支援（Physical Evidence）
2. 用戶行動（Customer Actions）
3. ── 互動線（Line of Interaction）──
4. 前台（Frontstage）：科技層 + 真人層
5. ── 可視線（Line of Visibility）──
6. 後台（Backstage）
7. ── 內部互動線（Line of Internal Interaction）──
8. 支援系統（Support Processes）

**流程階段（橫軸）**：根據服務特性設計，通常 4-6 個
- 建議命名格式：「動詞 + 核心動作」（如：發現、試用、使用、購買、分享）

**缺口診斷表格格式：**
| 缺口位置 | 問題描述 | 嚴重程度 | 來源依據 | 對應 SWOT | 建議方向 |
|---------|---------|---------|---------|---------|---------|

---

### 顧客旅程地圖（CJM）

**輸入**：Persona、服務藍圖的流程階段、真實用戶反饋
**核心問題**：Persona 從認知到使用的完整情緒旅程是什麼？
**輸出**：多欄位旅程表格 + 情緒分數明細表

**必要欄位（縱軸）：**
- 動機
- 行動
- 情緒曲線（含數字分數）
- 接觸點
- 感受 / 關鍵時刻
- 科技服務
- 行銷方法
- 目標

**情緒分數規則：**
- 每個分數必須對應到真實用戶反饋（媒體測試、App Store 評論等）
- 加分原因和減分原因都要列出，各自標來源
- Aha Moment 若有前提條件，要明確標注

**關鍵時刻（⚡）標記**：
- ⚡ = 中等摩擦點（有影響但可接受）
- ⚡⚡ = 嚴重摩擦點（可能導致用戶流失或核心需求無法被滿足）

**情緒分數明細表格式：**
| 階段 | 情緒分數 | 加分原因（含來源） | 減分原因（含來源） |
|------|---------|-----------------|-----------------|
